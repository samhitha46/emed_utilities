Organizer / Provider Workflow Architecture

Technical architecture for the conversational activity listing flow — from type detection through URL scraping, category confirmation, and submission to the verification queue. Scoped to the organizer workflow only.

Scope: This document covers only the provider/organizer listing flow as designed in the prototype. It does not cover HCP search and discovery, CDOI scoring, CMS/scraper pipelines, marketing services, or any other eMedEvents feature.
1
Three surfaces, one flow
Who uses what
Organizer portal
The conversational listing UI. Rendered as a Next.js app served from CDN. Holds all interaction state — the three-panel shell (nav, chat, checklist), drawer editor, and progress tracking. The organizer's browser is the only client that touches this flow.
API layer
A single FastAPI (Python) service. Receives all requests from the organizer portal. Orchestrates the LLM, scraper worker, database, and job queue. The frontend never calls external services directly — everything goes through this layer.
Admin review surface
A separate internal UI for eMedEvents staff. Shows the pending_review queue. Reviewers approve or flag submissions, add notes, and trigger notifications. Uses the same API as the organizer portal — different auth role, not a different system.
2
Step-by-step: what happens technically
From first description to submission confirmation
1
Opening question — does the organizer have a URL?
The first message asks a single binary question: does the organizer already have an established website or URL for their activity? No description, no type selection. The answer determines which of two paths the flow takes. No API call at this stage — pure client state.
Client onlyReact stateno API call
2A
Path A — URL provided: scrape + auto type detection
Before showing the URL input, the browser displays a scraping transparency notice: we only read publicly visible page content — the same information any visitor would see. Nothing behind a login, no private data. The organizer pastes their URL. The API first checks for an existing draft matching this URL and organizer (OQ-1 resume logic). If no draft found, it dispatches a Playwright scrape job. The scrape result is passed to the LLM which extracts structured fields and detects the activity type simultaneously — a single LLM call does both. The detected type and extracted fields are returned together. The browser shows a type confirm card: "Based on your page, this looks like an In-person conference — does that look right?" The left nav is not revealed until the organizer confirms the type.
POST /api/listings/scrape-url LLM: type detection + field extraction in one call SELECT existing draft by scraped_url + organizer_id Playwright headless browser SSE streaming to browser
2B
Path B — No URL: upsell, then free-text type detection
When the organizer has no URL, the flow presents a hosting upsell before continuing to listing. eMedEvents offers to build and host a dedicated event website. If the organizer is interested, their account email is pre-filled and confirmed — a lead is captured and routed internally. Whether interested or not, the flow transitions back to listing with a warm reassuring message. The organizer is then asked to describe their activity in their own words — no chip list, no forced categorisation. The description is sent to the LLM for type detection. If a type is identified with sufficient confidence, a confirm card is shown. If no clear type emerges, the flow defaults to In-person with an explanation that the type can be changed at any time from the dashboard.
Upsell lead → internal routing Pre-fill email from organizer account POST /api/detect-type LLM: type detection from free-text description Default: In-person if no type detected
3
Type confirmed — draft created in tbl_conferences
Once the organizer confirms the detected type, the API inserts a row into tbl_conferences with status=3 (draft) and conference_type set to the detected type integer. The auto-increment id becomes the draft_id for all subsequent calls. The left nav is revealed and populated based on the confirmed type. For Path A, scraped field data is already in tbl_listing_scrape_sessions and the flow moves directly into category confirmation. For Path B (no URL), the flow moves into manual field entry. Both paths converge from this point forward.
INSERT INTO tbl_conferences — status=3, conference_type=N Left nav revealed and populated Path A → category confirm with pre-filled fields Path B → manual field entry
4
Category-by-category confirmation (Path A) or field entry (Path B)
Path A: scraped fields are shown per category with confidence indicators. The organizer confirms or corrects each. Fields with high LLM confidence are pre-filled with a green dot. Low-confidence fields show an amber dot and an Add button — the organizer must actively enter these. Path B: the AI asks for each required field directly in the conversation. Each answer calls the field-level PATCH endpoint. Both paths write confirmed values to tbl_conferences and its related tables incrementally — the draft is updated on every category confirmation, not at the end.
PATCH /api/listings/{draft_id} UPDATE tbl_conferences SET ... WHERE id={draft_id} INSERT/UPDATE related tables as needed per category, incremental
5
Drawer edits
When the organizer clicks Edit or Add on a specific field, the drawer opens client-side with no API call. On save, the browser calls the API with the field key and new value. The backend validates the value — type check, format check, required field check — and returns either success or a specific error the UI displays inline in the drawer. No full-category re-save; this is a surgical field-level update.
PATCH /api/listings/{draft_id}/fields field-level validation in API returns: validation errors if any
6
Submit for review
Browser calls the submit endpoint. The API runs a completeness check — all required columns for the detected conference_type must be non-null. If the check fails, the API returns which fields are missing and the browser highlights them in the conversation. If it passes, the API sets status=4 (pending_admin_auth — your existing review state) on the tbl_conferences row, pushes a job to the SQS review queue, generates a reference number stored in conf_notes, and sends a confirmation email via SES. Reference number and estimated review time are returned to the browser.
POST /api/listings/{draft_id}/submit UPDATE tbl_conferences SET status=4 SQS — review queue SES — confirmation email reference stored in conf_notes
7
Review and decision (admin side)
The review job appears in the admin queue UI. A reviewer approves or flags the submission. On approval: status=1 (active) — listing becomes visible in the marketplace, organizer receives a notification email. On flag: status=3 (back to draft) — organizer receives an email with correction notes (stored in conf_notes) and a link to re-open their draft. After correcting, the organizer resubmits — status returns to 4 (pending_admin_auth) and re-enters the queue.
PATCH /api/listings/{id}/review (admin role) UPDATE tbl_conferences SET status=1 (approve) or status=3 + conf_notes (flag) SES — organizer notification
tbl_conferences status lifecycle (using your existing status integers):
3 — draft → 4 — pending_admin_auth → 1 — active or 3 — draft (flagged back) → 4 — pending_admin_auth
Your existing status values 2 (inactive), 5 (deleted), 6 (bulk_upload), 7 (expired) are unchanged and unaffected by this flow. No new status values are needed — the conversational flow maps entirely onto your existing enum.

New supplementary table — scrape session cache
tbl_listing_scrape_sessions (new)
Stores raw LLM extraction output temporarily during the organizer session. Separate from tbl_conferences because this data is transient — it is evicted once confirmed or after 2 hours, and never needs to be queried by the marketplace.

Columns:
conference_id (FK → tbl_conferences.id)
scraped_raw (JSON — full LLM field map with confidence)

expires_at (datetime — 2hrs from creation)
status (enum: pending / consumed / expired)
SQS review queue (existing or new)
review_jobs queue
Each submitted listing produces one SQS message: conference_id · submitted_at · organizer_email · conference_type. The admin review service polls this queue. Messages are retained until a reviewer takes action — SQS guarantees delivery even if the admin service restarts. Check if you already have an SQS queue on your AWS account before creating a new one.
Why not a separate listings table? Writing directly into tbl_conferences with status=3 means your existing admin tools, reports, and CMS already work with the new drafts without modification. Keeping drafts in a separate table would require syncing, merging, and updating every downstream system that reads tbl_conferences. The two new columns (scraped_url, submitted_at) are a minor schema change — an ALTER TABLE with nullable columns that has zero impact on existing rows or queries.
Action required — conference_type reconciliation: Your existing conference_type integers (1=Live Conference, 2=Text-Based CME, 3=Webcasts, 4=Journal CME, 5=Publications, 6=Others) need to be mapped to the 7 types in the conversational flow. In-Person and Hybrid are not currently distinct types in your schema. This needs a decision: add new type integers, or map In-Person and Hybrid both to type 1 (Live Conference) with a sub-type flag using the existing conference_sub_types column?
4
AI integration — the two LLM calls
What the prompts do and how they are structured
A
Path A — Combined scrape + type detection prompt
When the organizer provides a URL, a single LLM call handles both type detection and field extraction simultaneously. System prompt defines the 7 activity types with their signals, plus the full field schema for all types. User message is the cleaned scraped HTML plus the URL. The model returns a JSON object with two top-level keys: detected_type (type string + confidence 0–1) and fields (map of field name → value + confidence). This is more efficient than two separate calls and avoids the latency of waiting for type detection before starting field extraction. If type confidence is below 0.75, the confirm card shows the top two candidates and asks the organizer to choose.
System: 7 type definitions + full field schema User: cleaned HTML + URL Response: detected_type + fields[] in one JSON object Threshold: 0.75 confidence for type
B
Path B — Free-text type detection prompt
When the organizer has no URL, type detection runs on their free-text description of the activity. System prompt defines the 7 types with their signals. User message is the organizer's description in their own words. Response is JSON: type · confidence · reasoning. If confidence is below 0.75 the API returns the top two candidates and the frontend shows the type picker chips. If no type can be identified at all (organizer is still planning, no clear signals), the API returns type: null and the frontend defaults to In-person with an explanation that the type can be changed at any time from the dashboard. The reasoning field is logged for prompt improvement but never shown to the organizer.
System: 7 type definitions + signals User: free-text description Response: type · confidence · reasoning null type → default to In-person
C
Path B — Manual field entry (no URL)
When the organizer has no URL — whether they declined the upsell or simply don't have one — no scrape job is ever dispatched. After type is confirmed, the listing record is created with all fields null. The conversation flow asks for each required field directly via chat. Each answer calls the field-level PATCH endpoint. This is the same endpoint used for drawer edits — no special code path. The LLM is not involved in field collection in this path; the organizer's answers are taken at face value.
No scrape job PATCH /api/listings/{draft_id}/fields Same endpoint as drawer edit No LLM call in this path
Cost control: Each organizer session makes at most 2 LLM calls — one for type detection, one for field extraction. Type detection is fast and cheap (short prompt, short response). Field extraction is the expensive call (large HTML input). A per-organizer rate limit of 5 scrape attempts per session prevents abuse. All LLM calls are logged with input/output token counts for cost monitoring.

What you build vs what you buy
Technology decisions for a 1–3 person team
You write this code
  • Conversational UI — Next.js (React 18)
  • API service — Python / FastAPI
  • Scraper worker — Python + Playwright
  • Type detection prompt
  • Field extraction prompt (one per activity type)
  • Field validation logic
  • Admin review UI
  • SSE streaming from API to browser
AWS managed — you configure, not operate
  • RDS MySQL — existing instance, reused as-is
  • tbl_listing_scrape_sessions in MySQL — replaces need for Redis for this workflow
  • SQS — review job queue
  • SES — transactional email
  • S3 — file uploads (banners, brochures)
  • ECS Fargate — container hosting
  • Cognito — organizer + admin auth
  • CloudFront — CDN for frontend
External APIs — you call, they operate
  • Anthropic API — type detection + field extraction
  • No other external dependency for this specific workflow
Why FastAPI not PHP? The two requirements that PHP handles poorly for this specific workflow: (1) the scrape result needs to stream back to the browser as categories resolve — PHP's synchronous request model requires workarounds for SSE; (2) the scraper worker is a long-lived async process, not an HTTP request handler — PHP was not designed for this. Python handles both naturally. FastAPI connects to MySQL via aiomysql or SQLAlchemy + PyMySQL — your existing RDS MySQL instance is accessed directly, no migration required.

Decisions log
Resolved decisions and open items still pending
Resolved
OQ-1 ✓
Resume flow — decided. The organizer always starts as if creating a new listing. There is no "resume" prompt on entry. The draft detection happens at the URL step: when the organizer pastes a URL, the API checks tbl_conferences for an existing draft (status=3) owned by this organizer with a matching scraped_url or matching title extracted from the URL. If a match is found, the flow surfaces the existing draft and pre-populates all already-confirmed fields, skipping those categories entirely — the organizer is taken directly to the first incomplete section. If no match, a new draft is created as normal.
Technical implementation:
On POST /api/listings/scrape-url, before dispatching the scrape job, the API runs:
SELECT id, status FROM tbl_conferences WHERE created_by = {organizer_id} AND scraped_url = {url} AND status = 3 ORDER BY created_date DESC LIMIT 1
If a row is returned → load that draft and its already-confirmed fields, return to browser with draft_found: true and the sections already completed. The browser skips those sections in the conversation and advances the left nav dots to reflect completed state. If no row → proceed with new draft creation as normal.
OQ-2 ✓
Scrape failure fallback — decided. If the scraper cannot access the page (robots.txt block, bot detection, timeout, invalid URL), the flow drops silently to manual mode but the organizer is informed clearly. The message shown: "We weren't able to auto-fill from that page — some sites block automated access. No problem, we'll walk you through each field together." No data is lost, no error state. The conversation continues exactly as the no-URL path, with the AI asking for each required field in sequence. The organizer does not need to take any action beyond reading the message.
Technical implementation:
The scraper worker catches any fetch failure (HTTP error, timeout, robots.txt disallow, Playwright exception) and returns a scrape_status: blocked result to the API rather than throwing. The API sets the draft's scraped_url to the submitted URL (for audit) but proceeds as if no URL was provided. The SSE stream sends a scrape_failed event to the browser. The browser displays the friendly fallback message and transitions to manual field-by-field flow. No retry is attempted automatically — the organizer can paste a different URL if they wish.
OQ-3 ✓
File uploads — decided. File uploads (brochure, banner images) happen inside the conversational flow via the right-side drawer. When the organizer reaches the Collaterals & Branding category, the AI presents an "Add Brochure" action. Clicking it opens the drawer with a file upload input. The file is uploaded directly to S3 from the browser (using a pre-signed URL issued by the API), and the S3 key is written back to the relevant column in tbl_conferences or its related table on completion. No file data passes through the API server — the browser uploads directly to S3.
Technical implementation:
1. Organizer clicks "Add Brochure" → drawer opens with file input.
2. Browser calls GET /api/listings/{draft_id}/upload-url?type=brochure → API returns a pre-signed S3 PUT URL valid for 10 minutes.
3. Browser uploads file directly to S3 using the pre-signed URL.
4. On upload success, browser calls PATCH /api/listings/{draft_id}/fields with the S3 key → API writes it to the logo or relevant column. Drawer closes and the conversation confirms the upload.
